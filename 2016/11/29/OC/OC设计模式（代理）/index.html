OC设计模式（代理）

发表于 2016-11-29   |   分类于 Objective-C   |     |   阅读次数

在项目中我们经常会用到代理的设计模式，这是iOS中一种消息传递的方式，也可以通过这种方式来传递一些参数。

代理的基本使用

代理的组成

代理主要由三部分组成：

  • 协议：用来指定代理双方可以做什么

  • 代理：根据指定的协议，完成委托方需要实现的功能

  • 委托：根据指定的协议，指定代理去完成什么功能

下图阐述三方之间的关系：

实际应用中通过协议来规定代理双方的行为，协议中的内容一般都是方法列表，当然也可以定义属性。

协议是公共的定义，如果是某个类使用，就写在某个类中。如果是多个类用一个协议，可以写在一个Protocol文件。

协议只能定义公用的一套接口，类似于一个约束代理双方的作用。但不能提供具体的实现方法，实现方法需要代理对象去实现。协议可以继承其他协议，并且可以继承多个协议。

协议修饰符

@required 、@option,创建一个协议如果没有声明，则默认是@required。这两修饰符只是约定代理是否强制需要遵守协议，如果@required的方法没有遵守，会报一个黄色的警报。

代理的判断

无论是@option还是@required，在委托方调用代理方式时都要做一个判断，判断代理是否实现当前的方法，否则会导致崩溃。

if ([self.delegate respondsToSelector:@selector(userLoginWithUsername:password:)]) {
    [self.delegate userLoginWithUsername:self.username.text password:self.password.text];
}

代理的复用

在iOS中一个代理可以有多个委托方，而一个委托方也可以有多个代理。

代理对象在很多情况下是可以复用的，可以创建多个代理对象为多个委托方服务。

简单代理案例：

  • 首先定义一个协议类，定义公共协议
  • 代理方，实现具体登录流程，委托方不需要知道实现细节。

代理内存管理

为什么代理属性要设置为weak呢？

我们定义的指针默认都是strong类型的，而属性本质上也是一个成员变量和set、get方法构成的，strong类型的指针会造成强引用，必定会影响一个对象的生命周期，这也就会形成循环引用。

强引用

由于代理对象使用强引用指针，引用创建的委托方LoginVC对象，并且成为LoginVC的代理。这就会导致LoginVC的delegate属性强引用代理对象，导致循环引用的问题，最终两个对象都无法正常释放。

弱引用

将LoginVC对象的delegate属性，设置为弱引用属性。这样在代理对象生命周期存在时，可以正常为我们工作，如果代理对象被释放，委托方和代理对象都不会因为内存释放导致的Crash。

下面两种方式都是若引用代理对象，但是第一种在代理对象被释放后不会导致崩溃，而第二种会导致崩溃。

@property (nonatomic, weak) iddelegate;

@property (nonatomic, assign) iddelegate;

weak和assign是一种“非拥有关系”的指针，通过这两种修饰符修饰的指针变量，都不会改变被引用对象的引用计数。但是在一个对象呗释放后，weak会自动将指针置为nil，而assign则不会。在OC中，向nil发送消息是不会导致崩溃的，所以assign就会导致野指针的错误unrecognized selector sent to instance。

代理和block的选择

  • 多个消息传递，应该使用delegate。

  • 一个委托对象的代理属性只能有一个代理对象，如果想要委托对象调用对个代理对象的回调应该用block。

  • 单例对象最好不要用delegate。单例对象由于时钟都只是同一个对象，如果使用delegate，就会造成delegate属性被重新赋值的问题，最终只能有一个对象可以正常响应代理方法。

  • 代理更面向过程，block则更面向结果。

  • 从性能上说，block的性能消耗略大于delegate。
